# Agent 任务线程认领与交付 PRD

## 背景

AgentBean 的频道消息、讨论串和任务已经统一在 Team 内协作。用户希望 Agent 面对“任务式消息”时既能及时给出接单确认，又不把主频道刷成多段过程日志；真正的执行、结果和状态流转应该落在同一个任务线程里。

本 PRD 定义 Agent 处理任务或讨论串式消息的默认产品语义。

## 目标

- 让用户在发送可执行任务后立即知道任务已经被 Agent 接住。
- 区分普通问答和需要执行/产出的任务，避免所有消息都被转成任务。
- 保持主频道克制：短确认、必要进度、最终交付都围绕任务线程组织。
- Agent 交付后默认进入 `in_review`，等待用户审核；只有用户明确确认后才进入 `done`。

## 非目标

- 不引入新的任务状态枚举。
- 不让普通能力咨询、闲聊、寒暄自动变成任务。
- 不在本期实现复杂多步进度规划、自动阻塞诊断或用户确认语义的自然语言解析。

## 用户故事

1. 作为用户，我发送“请总结附件并产出报告”后，希望 Agent 先短回复“我来处理”，避免我以为没人响应。
2. 作为用户，我问“你能做什么”或“什么样的消息算任务”时，希望 Agent 直接回答，不创建任务、不认领。
3. 作为用户，我希望 Agent 的正式结果、附件、文件链接或验证说明出现在同一讨论串中，方便追踪上下文。
4. 作为用户，我希望 Agent 交付后任务进入待审核，而不是直接被系统标成已完成。

## 产品规则

### 任务判定

消息满足以下条件时才进入任务处理流程：

- 明确要求 Agent 执行、产出、处理附件、处理代码、查资料、生成文件、跑命令或验证结果。
- 或用户显式把消息标记为任务。
- 普通问答、能力咨询、寒暄、纯讨论不自动认领。

### 认领确认

任务被成功派发给 Agent 后，Agent 必须在线程里追加一条短确认消息：

> 我来处理，会先看请求和附件，再把结果发在线程里。

这条消息不是交付结果，只表示任务已经被 Agent 接住。

### 执行与进度

- Agent 后续正式执行下载附件、查资料、写内容、生成文件、跑命令或验证结果。
- 简单任务可以不发中途进度，直接交付。
- 耗时、分多步或可能阻塞的任务才发简短进度更新。

### 交付与状态

- Agent 正式结果仍作为 Agent 消息发回原任务线程。
- 派发成功时任务进入 `in_progress`。
- Agent 成功交付后任务进入 `in_review`。
- 用户明确确认 OK 后，任务才应被移动到 `done`。
- 失败或阻塞时不自动 `done`，应保留可继续处理的状态。

## 验收标准

- Task-like channel message 会创建任务线程、分配给被派发 Agent，并生成 Agent 接单确认消息。
- 接单确认通过 `channel:message` 实时推送给已订阅 Web client。
- Agent 成功返回结果后，结果消息进入同一线程，任务状态为 `in_review`。
- 普通问答仍只是普通消息/普通 dispatch，不产生任务认领消息。
- SQLite 持久化中的任务状态与内存 use case 返回值一致。

## 风险与边界

- 当前任务判定仍是轻量启发式，后续可引入更明确的 intent classifier 或用户可配置规则。
- 接单确认是固定短句，后续可以按附件、代码、资料等任务类型生成更贴切的短确认。
- 本期只定义“交付后待审核”；“用户说 OK 自动 done”的自然语言确认仍留给后续切片。
