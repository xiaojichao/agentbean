# 设备页：区分当前设备与远程设备 设计

> 日期：2026-07-09
> 分支：worktree-device-local-remote
> 关联记忆：[[agentbean-device-agent-ownership-auth]] / [[custom-agent-runtime-readonly-causes]] / [[web-next-test-conventions]] / [[feedback-tech-choice-follow-convention]] / [[feedback-isolated-fix-branch]]

## 一、问题

设备页不区分「用户当前正在使用的本机设备」与「用户名下的其他远程设备」。拥有者可在 web 远程管理自己名下所有设备（授权键 `ownerId === userId`，设计如此，非 bug），但 UI 缺少本机/远程的视觉区分，导致三类实际问题。

### 1.1 现象

1. 给远程设备添加自定义 Agent、点「浏览项目目录」时，弹出「等待选择」但目录选择对话框**永不弹出**，前端一直挂起直到超时。
2. 远程设备的删除等高危操作没有针对性护栏，用户容易误删不熟悉的远程设备。
3. 设备列表/详情没有任何「这台是我现在正在用的」视觉标识。

### 1.2 根因

**本机依赖的能力（GUI 目录选择器）被无差别套用到远程设备。** 目录浏览对有 `deviceId` 的设备走 `selectDirectory(deviceId)`，经 socket 转发给目标设备 daemon，由 daemon 调 `osascript`（macOS）弹原生目录选择窗。远程设备若以无头 / SSH / 后台 launchd 方式运行 daemon，**没有桌面 GUI 会话**，`osascript` 弹不出窗也不报错，daemon 挂起，前端 `emitWithAck` 等到超时。

次要根因：UI 缺少「当前设备」视觉标识——尽管系统已有精确的本机判定键 `device.isLocal`，但仅用于零散逻辑（page.tsx:713/761），未作呈现。

## 二、目标 / 非目标

### 2.1 目标

1. UI 标识本机设备（`[本机]` badge），让用户一眼区分「我正在用的这台」与名下其他远程设备。
2. 远程设备目录浏览不再挂起，降级为手动填写项目绝对路径。
3. 远程设备删除加额外护栏（输入设备名确认），防误删。

### 2.2 非目标

- **不改授权模型**：拥有者仍可远程管理自己名下所有设备（mesh 远程管理价值保留）。
- **不动 server-next / daemon**：纯前端 `web-next` 改动。
- **不对其他操作加约束**：改 agent 配置、扫描设备等可逆 / 无害操作维持现状。
- **不做 daemon 桌面会话能力探测**：YAGNI，远程目录浏览直接降级，不尝试智能检测。

## 三、方案：按 isLocal 分流

统一主轴：**同一布尔值 `device.isLocal` 驱动三处改动的呈现与可用性。** 新增 `lib/device-permissions.ts` 纯函数作为唯一「本机/远程判定」出口，UI 各处引用，避免判定散落。

### 3.1 决策记录（用户拍板）

- **D1 目录浏览 = 方案 A**：远程设备（`isLocal=false`）不展示浏览按钮，引导手动填绝对路径。理由：彻底消除挂起、纯前端、契合手动填路径已有兜底；远程「恰好有桌面会话」是少数，不值得为此保留可挂起的尝试。
- **D2 删除约束 = 范围-中**：仅远程设备删除要求输入设备名确认；本机维持现有确认；改配置 / 扫描等其他操作不加约束。理由：护栏只加在不可逆且后效复杂（牵连 daemon 重连复活，见 [[agentbean-device-delete-daemon-not-stopped]]）的删除上，不削弱远程管理。
- **D3 视觉标识 = 方案 3**：只标少数特殊的「本机」（绿色 `[本机]` badge，列表项 + 详情头部），不标远程（无 badge 即远程）。文案 `本机`（与 isLocal 语义及现有 UI 文案统一）。
- **D4 边界降级**：未做 `/device-login` 时所有设备 `isLocal=false`，全部按远程对待 + 顶部引导提示。接受。

### 3.2 视觉标识 `[本机]` badge

- 判定：`device.isLocal === true`。
- 样式：沿用设备列表已有的「可升级」badge（page.tsx:305 `rounded bg-amber-50 px-1 text-[10px] font-medium text-amber-700`），换绿色 `bg-green-50 text-green-700`。
- 位置：① 设备列表项设备名旁（page.tsx:300）；② `DeviceDetail` 详情头部。

### 3.3 目录浏览降级（方案 A）

- 改动 `DirectoryPickerButton`（page.tsx:106 `browse`），新增 `isLocal` 入参。
- `isLocal === false`（远程）：**不渲染浏览按钮**；项目目录输入框下方常驻提示「远程设备请直接填写该设备上的项目绝对路径」。
- `isLocal === true`（本机）：维持现状（走 `selectDirectory(deviceId)` → 本机 daemon 弹窗）。
- 无 `deviceId` 的浏览器原生 picker fallback（page.tsx:131）维持不变。

### 3.4 远程删除护栏（范围-中）

- 改动删除确认对话框（page.tsx:736-751）+ `confirmDeleteDevice`（416）。
- 目标设备 `isLocal === false`（远程）时追加：
  - 醒目提示「这是远程设备，删除不可撤销，且会影响该设备的 daemon 连接」。
  - 输入设备名输入框；「确认删除」按钮 disabled 由 `deviceDeleteSaving` 扩展为 `deviceDeleteSaving || (isRemote && typedName !== displayName)`。
- 本机设备（`isLocal === true`）维持现有「确定要删除设备 X 吗？此操作不可撤销。」确认。

## 四、判定键与数据流

`device.isLocal` 由后端 DTO 下发（`usecases.ts:4370` `isDeviceLocalToHint(device, currentDeviceId)`），`currentDeviceId` 来自 web socket auth 上报的本机设备 id（`socket-server.ts:754`，由 `/device-login` 流程写入 `getStoredDeviceId`）。前端直接读取 `device.isLocal`，**无需新增接口、无需改后端**。三处改动共用同一判定。

边界（`usecases.ts:4338`）：`currentDeviceId` 为空（未 device-login）时所有设备 `isLocal` fail-closed 为 false。

## 五、边界处理（未 device-login）

当设备列表中**不存在任何** `isLocal===true` 的设备（典型：纯账号密码登录未做 device-login）→ 无任何 `[本机]` 标识，全部设备按远程对待（目录浏览全降级、删除全需输设备名）。此时在设备页顶部显示提示 + 链接：

> 未关联本机设备，目录浏览等操作将按远程设备处理。[前往设备登录]

链接指向 `/device-login`。

## 七、测试策略

遵循 web-next 惯例（[[web-next-test-conventions]]）：`tests/` 只测 lib 纯函数、不测 page.tsx 组件、无 jsdom。判定逻辑全部抽到 `lib/device-permissions.ts` 并单测：

- `canBrowseDirectory(isLocal)`：本机 true、远程 false。
- `requiresDeleteNameConfirm(isLocal)`：本机 false、远程 true。
- `hasLocalDevice(devices)`：含 isLocal 设备 true、全远程 false、空列表 false。

## 八、范围

纯前端 `web-next`。授权模型、server-next、daemon 均不动。
