# AgentBean 项目任务与文件管理架构设计

- 日期：2026-07-17
- 目的：说明产品模型、领域边界、关键流程和验收标准，供实现同学据此拆解开发任务。
- 范围：频道项目化、阶段任务、项目产物、文件版本、最终版、多文件选择、人审与 Agent 交接。

---

## 1. 结论

首版项目管理不新增一个脱离频道的顶层 `Project` 系统。一个频道就是一个项目空间，频道继续承载成员、权限、消息、任务线程、Agent 可见性和文件可见性。

核心决策：

- **频道即项目**：频道天然具备项目能力，不需要用户手动开启“项目模式”，也不新增项目容器。
- **阶段不是状态列**：剧本、服装、分镜、审核是项目阶段，应该绑定到任务图节点；不要把它们塞进 `TaskStatus`。
- **文件不是普通附件**：聊天附件只是文件出现的位置；项目产物需要有类型、版本、审核态、来源和最终版指针。
- **最终版不是文件名或标签**：最终版必须是 Server 持久化的版本指针，可追溯、可审计。
- **聊天不是事实源**：聊天用于协作和触发操作；项目阶段、审核、版本和最终版都必须由 Server 写入事实。

一句话定义：

> 频道是项目空间，阶段是有依赖和审核规则的任务节点，文件是可版本化、可审核、可设最终版的项目产物。

## 2. 用户流程

目标用户流程如下：

1. 真人负责人创建或进入一个频道，这个频道就是项目协作空间。
2. 项目里有真人负责人、真人审核者，以及多个 Agent。
3. 不同 Agent 负责不同阶段，例如编剧、服装、分镜。
4. 编剧 Agent 先生成剧本、提示词、参考图等多个文件。
5. 真人审核这些文件，可能通过、拒绝、要求修改，也可能选定最终版。
6. 审核通过后，后续 Agent 基于已通过或最终版文件继续工作。
7. 如果一条消息里有多个文件，用户必须能明确选择其中第 3 个文件或多个文件，发起修改、审核、交给 Agent 或设为最终版。

这个流程的关键不是“任务列表更好看”，而是让项目负责人始终知道：

- 当前项目有哪些阶段。
- 每个阶段由谁负责。
- 哪些阶段正在执行、等待审核、已经完成或被阻塞。
- 每个阶段产出了哪些文件。
- 哪些文件是草稿、候选、已通过、被拒绝或最终版。
- 后续 Agent 使用的是哪一版输入。

## 3. 目标与非目标

### 目标

- 在任务页清楚展示项目阶段、负责人、执行 Agent、审核者、状态、阻塞点和交付物。
- 支持“Agent 产出 -> 真人审核 -> 后续 Agent 继续”的顺序推进。
- 支持 Agent 输出物按脚本、提示词、图片、分镜、服装参考等类别归档。
- 支持多文件消息中的精确选择和修改指派。
- 支持可追溯的最终版选择。
- 复用现有 `Channel`、`TaskStatus`、Task DAG、`ManagementRun`、`AgentInvocation`、`Artifact` 和 `WorkspaceRun`。

### 非目标

- 首版不做跨频道项目。
- 首版不做甘特图、预算、人力负载、资源排期。
- 不用业务阶段扩展 `TaskStatus`。
- 不要求所有外部 Agent 直接理解项目数据库。
- 不用文件名、标签或目录名判断最终版。
- 不允许外部 Agent 直接修改项目事实；外部 Agent 只能产出结果或 proposal，由 Manager/Server 校验后写入事实。

## 4. 核心模型

本文只定义职责和关系，不把字段写死成最终 DTO 或表结构。实现同学可以按现有代码风格调整命名和拆分，但语义边界不要改变。

### 4.1 ChannelProjectProfile

频道项目画像是内部配置和投影，不是用户可感知的“模式开关”。所有频道都天然可以承载项目；当频道出现项目负责人、阶段、审核规则或项目产物时，Server 可以按需创建或更新这类记录。

它负责表达：

- 项目状态是进行中、暂停、完成还是归档。
- 项目总负责人是谁。
- 默认审核者是谁。
- 是否使用某个阶段模板。
- 默认输出归档策略是什么。

它不替代 `Channel`，也不决定频道是不是项目。频道名称、成员、权限、可见性、归档状态仍由 `Channel` 管理。

### 4.2 ProjectStage

项目阶段，例如剧本、服装、分镜、总审。

它负责表达：

- 阶段目标是什么。
- 阶段绑定哪个任务或任务线程。
- 阶段负责人是真人还是 Agent。
- 阶段是否需要审核，谁审核。
- 阶段期望产出哪些类型的文件。

阶段状态默认从绑定任务和产物审核状态聚合，不新增业务状态枚举。`todo`、`in_progress`、`in_review`、`done`、`closed` 仍是基础任务状态。

### 4.3 ProjectStageEdge

阶段之间的依赖关系。

它负责表达：

- 哪个阶段完成后才能启动后续阶段。
- 哪个阶段的产物会作为后续阶段上下文。
- 哪些产物是后续阶段必需输入。

典型关系：

- 剧本通过后，服装阶段才可以开始。
- 剧本最终版作为分镜 Agent 的输入。
- 分镜和服装都完成后，进入总审。

### 4.4 ProjectArtifactCollection

项目产物集合，表示一个逻辑交付物，例如“第 1 集剧本”“女主服装参考”“分镜图组”。

它负责表达：

- 这个产物属于哪个项目、阶段和任务。
- 产物类型是什么。
- 当前版本是哪一个。
- 最终版是哪一个。
- 当前审核状态是什么。

集合不保存文件内容。文件内容仍由已有 `Artifact` 管理。

### 4.5 ProjectArtifactVersion

项目产物版本，绑定一个已有 `Artifact`。

它负责表达：

- 这是某个产物集合的第几个版本。
- 文件来自哪条消息、哪次 Workspace Run、哪次 Agent invocation、哪个任务。
- 这个版本是草稿、候选、待审核、已通过、已拒绝、最终版还是归档。
- 它从哪些旧版本或输入演进而来。

`Artifact` 是文件事实；`ProjectArtifactVersion` 是项目语义事实。一个文件可以只是普通附件，也可以被提升为项目产物版本。

### 4.6 ArtifactReview

审核记录。审核对象可以是阶段、产物集合或具体产物版本。

它负责表达：

- 谁审核了什么。
- 审核结论是通过、拒绝还是要求修改。
- 审核意见对应哪条消息。
- 审核依据引用了哪些文件版本。

审核通过不等于最终版。最终版选择是单独动作。

### 4.7 ArtifactReference

聊天、任务和 Agent invocation 中使用的稳定文件引用。

它负责表达：

- 用户选中的到底是哪一个文件或哪一个版本。
- 这个引用在当前消息里如何展示，例如 `F3`、`IMG-02`、`SCRIPT-v4`。
- 这个引用来自哪条消息和消息内第几个文件。

消息内序号只能帮助用户理解当前上下文，不能作为长期身份。长期身份必须依赖 `artifactId` 或 `artifactVersionId`。

## 5. 关键规则

### 5.1 阶段推进

推荐默认流程：

1. 项目负责人在频道中创建阶段或指定项目负责人，Server 按需补齐项目画像。
2. Server 根据模板或手动配置生成阶段。
3. 某个 Agent 开始负责一个阶段，绑定任务进入 `in_progress`。
4. Agent 交付文件后，阶段进入 `in_review`，相关产物版本进入待审核。
5. 真人审核通过后，阶段可进入 `done`，产物版本进入已通过。
6. 项目负责人或审核者可以把某个已通过版本设为最终版。
7. 后续阶段启动时，Server 把已通过或最终版产物作为上下文传给下一个 Agent。

如果审核拒绝，阶段回到可继续执行状态，并记录被拒绝的具体版本和审核意见。后续修改必须引用被拒绝版本，不能只靠自然语言说“改一下刚才那个”。

### 5.2 文件归档

Agent 仍然把文件写入现有 `AGENTBEAN_OUTPUT_DIR`。当频道、阶段或单次 invocation 配置了输出分类策略时，可以下发这些约定目录，例如：

```text
outputs/
  scripts/
  prompts/
  images/
  storyboards/
```

Device 不需要理解项目数据库，只要按约定目录输出文件。Artifact collector 上传后，由 Server 根据输出路径、mime type、阶段期望和 invocation context 判断产物类型，并创建或更新项目产物集合和版本。

### 5.3 最终版

最终版必须满足：

- 属于当前频道的项目产物集合。
- 属于可最终化的产物集合。
- 通常应来自已通过版本。
- 设置动作由有权限的人类触发，或由 Manager 代表明确的人类确认触发。

同一产物集合同时只能有一个最终版。新最终版产生时，旧最终版不删除，应保留历史并回到已通过或被取代状态。

最终版不能只靠这些方式表达：

- 文件名含 `final`。
- 标签含“最终版”。
- 放在某个 `final/` 目录。
- 聊天里用户说“这个就是最终版”但 Server 没有写入事实。

### 5.4 `pathKind` 不承载业务语义

现有 `Artifact.pathKind` 继续只表达文件来源，例如 `upload`、`workspace`、`generated`。

项目产物类型必须放在项目产物元数据中，例如 `script`、`prompt`、`image`、`storyboard`、`costume_reference`。

不要把“文件从哪里来”和“文件在项目里代表什么”混在一个字段里。

### 5.5 多文件选择

当一条消息包含多个文件时，UI 可以展示短编号：

- `F1`、`F2`、`F3`：当前消息内的编号。
- `IMG-02`、`SCRIPT-v4`：更稳定的项目语义编号。

用户点击或选择文件后，发送消息、审核、修改、交接时必须携带稳定引用，例如 `artifactVersionId`。Agent invocation 不应该接收“第 3 个文件”这种模糊输入。

如果用户说“改第 3 个”，系统可以在当前焦点消息内解析并插入引用 chip。如果当前上下文里有多个可能的“第 3 个”，必须让用户选择，不能猜。

## 6. 人审与 Agent 交接

阶段可以配置审核策略：

- `none`：交付后可自动推进。
- `human`：必须真人审核后推进。
- `manager`：Manager 可根据验收标准和 evidence 判定，必要时转真人。

创意类项目里，剧本、分镜、最终合成等阶段默认应该走真人审核。

后续 Agent 的输入由 Server 组装，不靠复制完整聊天记录。输入应包含：

- 上游阶段摘要。
- 上游阶段已通过或最终版产物。
- 审核意见。
- 当前阶段目标和验收标准。
- 必需附件和文件引用。
- 必要的线程消息片段。

沿用多 Agent 交接设计的安全边界：

- 外部 Agent 可以提出交接 proposal。
- Manager/Server 校验阶段依赖、审核状态、目标 Agent 可见性和权限。
- 校验通过后才创建正式 handoff 或后续 invocation。
- 目标 Agent accepted/running 后，阶段 owner 或 active owner 才切换。
- 目标 Agent 未接收时，当前阶段仍归 source Agent 或项目负责人。

## 7. 产品界面期望

### 7.1 项目总览

任务页进入频道时，不需要用户先切换“项目模式”。如果该频道已有阶段、项目产物或项目负责人，默认展示阶段总览；如果还没有这些项目数据，则可以保持普通任务视图，并引导用户创建第一个阶段。

阶段卡片应让项目负责人一眼看到：

- 阶段名称。
- 当前状态。
- 负责人。
- 审核者。
- 前置依赖是否满足。
- 产物数量和最终版数量。
- 当前阻塞原因。

普通任务看板仍可保留，作为状态视图或执行者视图。

### 7.2 阶段详情

点击阶段后，应看到：

- 阶段目标和验收标准。
- 绑定任务线程。
- 当前负责人和 reviewer。
- 上游输入产物。
- 当前阶段产物集合和版本。
- Agent 调用记录和 Workspace Run。
- 审核历史。
- 通过审核、要求修改、重新生成、设为最终版、交给下一阶段等动作。

### 7.3 文件库

项目文件库应按逻辑产物集合组织，而不是按聊天消息附件堆叠。

每个集合应展示：

- 产物名称和类型。
- 所属阶段。
- 当前版本。
- 最终版。
- 审核状态。
- 历史版本。
- 来源消息、任务或 Workspace Run。

### 7.4 聊天消息

聊天中的文件卡片应支持：

- 显示项目语义编号和版本。
- 单选或多选。
- 作为修改对象引用。
- 作为审核对象引用。
- 设为最终版。
- 交给指定 Agent 或阶段。

聊天负责触发操作，操作结果必须落回项目事实模型。

## 8. 权限与安全

- 只有能访问频道的人才能看到项目阶段和项目文件。
- 只有项目负责人、阶段 reviewer 或被授权 Manager 才能通过审核或设最终版。
- Agent 可以产出文件和建议下一步，但不能直接批准审核、设最终版或改项目阶段事实。
- 所有跨阶段交接都必须校验目标 Agent 可见性、频道权限和阶段依赖。
- 被归档或删除的频道不能继续产生项目变更。

## 9. 验收标准

实现完成后，至少应满足这些场景。

### 场景 1：项目负责人看阶段进度

项目负责人进入频道任务页，可以看到剧本、服装、分镜、审核等阶段，以及每个阶段的负责人、状态、审核者、阻塞点和交付物数量。

### 场景 2：Agent 产出后进入审核

编剧 Agent 在剧本阶段产出多个文件后，系统能把这些文件归档为项目产物版本，并让阶段进入待审核状态。

### 场景 3：多文件中选择指定文件修改

一条 Agent 消息包含 10 个文件时，用户可以明确选择第 3 个文件发起修改。发送给 Agent 的请求必须引用稳定文件版本，而不是只写“第 3 个”。

### 场景 4：最终版可追溯

用户把某个已通过版本设为最终版后，文件库能显示最终版；后续能追溯它来自哪条消息、哪次运行、哪个阶段和哪个 Agent。

### 场景 5：审核通过后交给下一阶段

剧本阶段审核通过后，服装或分镜 Agent 能拿到已通过或最终版剧本作为输入继续工作。

### 场景 6：审核拒绝后重新生成

审核者拒绝某个版本并提出修改意见后，系统保留拒绝记录，并能基于被拒绝版本发起重新生成或修改请求。

## 10. 暂不解决

这些能力不放进首版：

- 跨频道项目。
- 项目模板市场。
- 甘特图、日历排期和资源负载。
- 文件内容 diff。
- 多最终版分支管理。
- 面向外部客户的交付门户。

首版先把“项目阶段、审核、文件版本、最终版、多文件选择、Agent 交接输入”做对。
